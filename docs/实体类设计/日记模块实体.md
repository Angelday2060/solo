# 日记模块 — 实体类设计

依据《solo-需求文档》4.1 节与产品原型中的「按日聚合列表 · 正文 · 元数据」约定。与前端 TypeScript、Rust `serde`、SQLite 表结构应对齐字段命名（建议使用 `snake_case` 落库、`camelCase` 仅存 TS 层时在边界转换）。

**正文形态**：日记**仅使用纯文本**。需求文档中的「富文本或 Markdown」在本模块实现上收窄为多行纯文本（无存储层格式枚举）；若日后支持富文本/Markdown，需另增格式字段并做数据迁移。

---

## 1. 正文约定

- `body` 存 **UTF-8 纯文本**：不按 Markdown/HTML 解析；入库与检索均为用户输入的字面文本。
- 换行使用实际换行符（如 `\n`）；检索与展示按字面文本处理。
- 界面侧使用多行文本编辑器即可。

---

## 2. 日记条目 `DiaryEntry`

核心业务实体，对应「某日之下的一条日记记录」（同日可多条，列表按日期聚合展示）。

| 属性 | 类型 | 必填 | 约束 / 说明 |
|------|------|------|-------------|
| `id` | string | 是 | 主键，稳定不可变（如 UUID） |
| `entry_date` | string | 是 | **归属日历日**，`YYYY-MM-DD`，与侧栏/列表「按日聚合」一致 |
| `title` | string \| null | 否 | 自定义标题；`null` 或空串按产品文案展示为「无标题」 |
| `body` | string | 否 | **纯文本**正文；首期允许为空草稿，删除策略由产品确认 |
| `weather` | string \| null | 否 | 可选元数据（如「晴」「小雨」）；设置中可关闭采集 |
| `mood` | string \| null | 否 | 可选元数据（如心情枚举键或简短文案）；设置中可关闭采集 |
| `sort_order` | number | 否 | **同一 `entry_date` 内**排序，越小越靠前；单条同日可不使用 |
| `created_at` | string | 否 | ISO 8601，首次保存时间 |
| `updated_at` | string | 否 | ISO 8601，最后修改时间 |

**正文体积**：超长文本建议在实现层约定单行上限或改用外链存储策略，并在《开发说明》中单独立条；本文仅定义字段语义。

**与「一篇一日」模型的关系**：若产品收紧为「每个自然日仅一篇」，可在库层对 `(entry_date)` 或 `(entry_date, user_id)` 加唯一约束；当前模型默认支持**同日多篇**，由 `sort_order` 排序。

---

## 3. 标签（可选扩展）`DiaryTag` / 关联

需求 4.1 提及「标签（若支持）过滤」。首期可不落库；若实现，建议：

### 3.1 标签定义 `DiaryTag`

| 属性 | 类型 | 必填 | 约束 / 说明 |
|------|------|------|-------------|
| `id` | string | 是 | 主键 |
| `name` | string | 是 | 展示名，唯一（或与规范化名唯一） |
| `created_at` | string | 否 | ISO 8601 |

### 3.2 条目—标签关联 `DiaryEntryTag`

| 属性 | 类型 | 必填 | 约束 / 说明 |
|------|------|------|-------------|
| `entry_id` | string | 是 | `DiaryEntry.id` |
| `tag_id` | string | 是 | `DiaryTag.id` |

主键可为 `(entry_id, tag_id)`。

---

## 4. 实体关系（提要）

- **必选**：独立实体仅 **`DiaryEntry`**（MVP）。
- **可选**：`DiaryEntry` ←→ `DiaryTag`（多对多，经 `DiaryEntryTag`）。
- **跨模块**：笔记「链接到某条日记」属扩展字段（如在笔记侧存 `diary_entry_id`），本文不展开笔记实体。

按日期范围、关键词的检索由查询产生；日历视图聚合不作为独立持久化实体类。
